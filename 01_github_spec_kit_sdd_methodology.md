# Part 1 — 배경과 문제 인식

---

## 1. 시대적 배경: AI가 코딩 방식을 바꾸다

SDD를 이해하려면 먼저 AI 시대에 개발 방식이 어떻게 변화해 왔는지를 살펴볼 필요가 있습니다.

---

### 바이브 코딩(Vibe Coding)의 등장

2025년 2월, OpenAI 공동 창업자이자 전 Tesla AI 총괄 책임자인 **안드레이 카파시(Andrej Karpathy)** 는 소셜미디어(X)에 짧은 글을 올렸습니다.

> *"There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists."*
>
> "나는 이것을 '바이브 코딩'이라고 부른다. 분위기에 완전히 몸을 맡기고, 지수적 성장을 받아들이며, 코드가 존재한다는 사실조차 잊어버리는 것이다."

이 글은 전 세계 개발자 커뮤니티에 즉각적인 반향을 일으켰습니다. 바이브 코딩이란, 개발자가 AI에게 자연어로 원하는 것을 말하면 AI가 코드를 생성해 주는 방식입니다. 개발자는 코드를 한 줄씩 작성하는 대신, **무엇을 만들지(What)** 를 설명하고 AI가 **어떻게 만들지(How)** 를 결정하는 구조입니다.

```
[기존 방식]   개발자 → 직접 코드 작성 → 실행
[바이브 코딩] 개발자 → 자연어 지시 → AI가 코드 생성 → 실행
```

초기에는 주로 개인 프로젝트나 프로토타이핑에 활용되었습니다. 복잡한 기술 지식 없이도 동작하는 소프트웨어를 빠르게 만들 수 있다는 점에서 큰 인기를 끌었습니다. 실제로 Y Combinator의 2025년 동계 배치(Winter 2025)에 참여한 스타트업 중 **21%가 전체 코드베이스의 91% 이상을 AI로 생성**했다고 알려져 있습니다.

> **💡 핵심 개념**
>
> 바이브 코딩은 AI가 코딩의 진입 장벽을 획기적으로 낮춰준 혁신이었습니다. 비개발자도 소프트웨어를 만들 수 있게 되었고, 개발자들은 반복적인 코드 작성에서 해방되기 시작했습니다.

---

### 바이브 코딩의 한계

그러나 바이브 코딩이 빠르게 확산되면서 한계도 분명하게 드러났습니다.

**문제 1: Context Rot (맥락 부식)**

```
[바이브코딩 시나리오]

1단계: "데미지 시스템 만들어줘"
       → 방어력 공식, 크리티컬 배율 설계  ✅

2단계: "이번엔 적 AI 만들어줘"
       → AI는 방금 전 데미지 시스템의 존재를 모름  ❌
       → 방어력 공식과 맞지 않는 별도 공격 로직 생성

3단계: "UI에 데미지 표시해줘"
       → 또 다른 데미지 계산 로직이 등장  ❌❌

결과: 각 작업은 멀쩡하지만, 전체를 합치면 엉망.
```

> **Context Rot (맥락 부식)**: 작업이 전환될 때마다 이전 맥락이 증발하여 코드 일관성이 붕괴되는 현상.

**문제 2: 품질과 일관성 부족** — AI가 생성한 코드가 당장 동작하더라도, 전체 시스템과의 정합성이나 장기적인 유지보수를 고려하지 않는 경우가 많았습니다.

**문제 3: 추적 불가능한 코드베이스** — 바이브 코딩으로 빠르게 생성된 코드는 왜 이런 구조로 만들어졌는지 근거가 없었습니다.

**문제 4: 프로덕션 수준에 미치지 못하는 품질** — 보안, 성능, 규정 준수 등 엔터프라이즈 환경의 요구 사항을 만족시키기 위해서는 단순히 '동작하는 코드' 이상이 필요했습니다.

---

### 에이전틱 엔지니어링(Agentic Engineering)으로의 진화

2026년 초, 카파시는 바이브 코딩의 후계 개념으로 **에이전틱 엔지니어링(Agentic Engineering)** 을 제시했습니다.

> *"'Agentic' because the new default is that you are not writing the code directly 99% of the time, you are orchestrating agents who do and acting as oversight."*

| 구분 | 바이브 코딩 (2025) | 에이전틱 엔지니어링 (2026~) |
|---|---|---|
| **주요 사용자** | 비개발자, 초보자, 빠른 프로토타이핑 | 전문 개발자, 엔터프라이즈 팀 |
| **AI의 역할** | 코드 생성기 | 자율 에이전트 |
| **개발자의 역할** | 지시자 | 조율자(Orchestrator) + 감독자(Oversight) |
| **품질 기준** | 일단 동작하면 성공 | 프로덕션 품질 유지 |
| **핵심 스킬** | 자연어 지시 | 구조 설계 + 에이전트 관리 |

---

### Software 3.0과 SDD의 연결고리

카파시는 이러한 흐름을 **Software 3.0** 시대로 설명합니다.

```
Software 1.0  →  개발자가 직접 명시적인 코드 작성
Software 2.0  →  데이터와 학습을 통해 모델이 동작 결정 (머신러닝)
Software 3.0  →  자연어 지시로 AI가 소프트웨어 생성 및 운영
```

개발 방법론의 역사적 진화도 함께 이해하면 도움이 됩니다.

```
전통적 개발:   Think → Code → Document
               (코드가 왕, 문서는 부산물)

TDD:           Test → Code
               (테스트가 먼저, 코드는 테스트를 통과하도록)

SDD:           Specify → Generate → Validate
               (명세가 왕, 코드는 명세의 표현)
```

> **🔑 핵심 메시지**
>
> 바이브 코딩이 AI 시대의 가능성을 열었다면, 에이전틱 엔지니어링은 그 가능성을 프로덕션 수준의 품질로 끌어올리는 방법론입니다. 그리고 **SDD는 에이전틱 엔지니어링을 실천하기 위한 구체적인 프레임워크**입니다.

## 2. AI 코딩의 현실

AI 코딩의 진짜 문제가 무엇인지를 살펴봅니다.

---

### 2.1 "그냥 시키면 되지 않나?" — 자유 형식 지시의 한계

> **"그냥 말로 시키면 되는 거 아닌가?"**

이 직관은 절반은 맞고 절반은 틀립니다. **자유 형식(Free-form) 지시만으로는 일관된 결과를 보장할 수 없습니다.**

```
개발자: "로그인 기능 만들어 줘."

AI가 가정해야 하는 것들:
├── 어떤 인증 방식? (JWT? 세션? OAuth?)
├── 어떤 데이터베이스? (MySQL? PostgreSQL? MongoDB?)
├── 비밀번호 해싱 방식? (bcrypt? SHA-256?)
├── 실패 시 처리? (잠금? 재시도 제한?)
├── 어떤 프레임워크? (FastAPI? Django? Express?)
└── 어떤 보안 수준? (금융급? 일반 웹?)
```

AI는 이 모든 것을 **스스로 가정(Assumption)** 합니다.

#### 반복적인 수정 루프

```
[1회] "로그인 기능 만들어 줘"
      → AI가 세션 기반으로 구현 (JWT를 원했는데)

[2회] "JWT로 바꿔줘"
      → AI가 refresh token 없이 구현

[3회] "refresh token도 추가해줘"
      → AI가 이전 코드와 충돌하는 새 코드 생성

[4회] "이전 코드와 합쳐줘"
      → AI가 컨텍스트를 잃고 다시 처음부터 시작...
```

이 루프에서 빠져나오려면 평균 **5~15회의 추가 대화**가 필요합니다.

> **💡 핵심 교훈**: AI는 마법이 아닙니다. 입력이 모호하면 출력도 모호해집니다.

---

### 2.2 "더 많이 알려주면 되지 않나?" — 정보 과부하의 역설

> **"그럼 더 자세하게, 더 많은 정보를 주면 되겠네?"**

**정보의 양이 많다고 해서 AI가 더 잘 이해하는 것은 아닙니다.**

#### "Lost in the Middle" 현상

AI는 긴 텍스트를 처리할 때 처음과 끝 부분을 잘 처리하지만, **중간 부분의 중요한 제약사항과 비즈니스 규칙은 상대적으로 누락**하는 경향이 있습니다.

```
긴 프롬프트:
┌──────────────────────────────────────────┐
│ [앞부분] 잘 처리됨 ✅                      │
│ [중간부분] 중요한 제약사항, 비즈니스 규칙 ⚠️ │
│ [끝부분] 잘 처리됨 ✅                      │
└──────────────────────────────────────────┘
```

| 상황 | 기대 | 실제 결과 |
|---|---|---|
| 짧은 지시 | 빠를 것 같다 | AI가 스스로 가정하여 일관되지 않은 결과 |
| 긴 비구조화 정보 | 더 잘 알겠지 | AI가 중요도 판단을 못해 핵심 요구사항 누락 |
| 구조화된 명세 | 번거롭다 | AI가 정확하게 이해하고 예측 가능한 결과 생성 |

> **💡 핵심 교훈**: AI에게 필요한 것은 **정보의 양**이 아니라 **정보의 구조**입니다.

---

### 2.3 진짜 문제: 커뮤니케이션 구조의 부재

> **문제의 본질은 커뮤니케이션 구조의 부재입니다.**

```
[인간 간 커뮤니케이션]
시니어 → 주니어: 조직 문화, 코딩 컨벤션, 도메인 지식을 공유
                  암묵적 지식 + 명시적 지시의 혼합

[AI와의 커뮤니케이션]
개발자 → AI: 공유하는 맥락이 없음. 모든 것을 명시적으로 구조화해야 함
```

```
커뮤니케이션 구조 부재
        │
        ├── AI가 임의로 기술 스택 선택 → 조직 표준과 불일치
        ├── AI가 임의로 보안 수준 결정 → 규정 준수 실패
        ├── AI가 임의로 아키텍처 패턴 선택 → 기존 시스템과 불일치
        └── 세션마다 다른 결과 → 일관성 없는 코드베이스
```

SDD가 제공하는 커뮤니케이션 구조는 다음과 같습니다.

```
constitution.md → "우리 팀은 이렇게 개발합니다" (공통 규칙)
spec.md         → "이 기능은 이렇게 동작해야 합니다" (명세)
plan.md         → "이렇게 구현할 것입니다" (설계)
tasks.md        → "이 순서로 작업합니다" (실행 계획)
```

> **🔑 핵심 메시지**: "좋은 명세는 좋은 코드보다 먼저 존재한다."

# Part 2 — SDD 이론

## 3. SDD란 무엇인가?

---

### 3.1 SDD의 정의와 등장 배경

#### Birgitta Böckeler의 분석

사양 기반 개발(Specification-Driven Development, SDD)을 가장 체계적으로 정리한 인물 중 한 명은 **Birgitta Böckeler** 입니다. 그녀는 전 세계 45개국 1만 명 이상의 기술 전문가 조직인 **Thoughtworks의 Distinguished Engineer(수석 기술자)** 입니다.

Böckeler는 소프트웨어 아키텍처의 세계적 권위자인 **Martin Fowler**가 운영하는 **martinfowler.com**에 SDD에 관한 분석 글을 게재했습니다.

> **martinfowler.com이란?**
> Martin Fowler는 『Refactoring』 등의 저서로 유명한 소프트웨어 개발의 선구자입니다. 그의 웹사이트는 소프트웨어 개발 방법론에 관한 가장 권위 있는 블로그 중 하나입니다.

Böckeler의 분석은 SDD를 **"AI 에이전트가 단순한 코드 생성기를 넘어 실제 개발 파트너로 기능하게 만드는 방법론"** 으로 정의합니다.

#### SDD의 핵심 정의

> **"코드보다 사양(Specification)을 먼저 정의하라. 사양이 명확하면, AI는 올바른 코드를 만들 수 있다."**

```
[기존 패러다임]
요구사항(말로) → 설계(머릿속) → 코드(실체) ← 진실의 근원
                                    ↑
                     문서는 코드를 따라가며 구식이 됨

[SDD 패러다임]
사양(문서) ← 진실의 근원 → AI가 코드로 변환
    ↑                              ↓
코드 변경 시 사양 먼저 수정      코드는 사양의 표현
```

---

### 3.2 기존 개발 방식과 SDD의 차이

| 구분 | 기존 방식 | SDD |
|---|---|---|
| **진실의 근원** | 코드 | 사양 |
| **사양의 역할** | 코드를 만들기 위한 수단 | 핵심 아티팩트 |
| **코드의 역할** | 최종 결과물 | 사양의 구현체 |
| **유지보수 대상** | 코드 | 사양 + 코드 |
| **AI 활용 방식** | 코드 생성 보조 도구 | 사양을 코드로 번역하는 실행자 |
| **변경 시작점** | 코드 수정 | 사양 수정 → 코드 재생성 |
| **문서 상태** | 코드와 점점 괴리됨 | 코드와 항상 동기화 |

---

### 3.3 SDD의 3가지 구현 레벨

Böckeler는 SDD를 성숙도에 따른 **3가지 레벨**로 구분했습니다.

```
코드 중심 ◄──────────────────────────────────────► 명세 중심
          │                   │                   │
     Spec-First          Spec-Anchored       Spec-as-Source
    (명세 우선)          (명세 고정)          (명세가 원천)
      낮은 규율           중간 규율            높은 규율
```

---

#### Level 1: Spec-First (명세 우선)

```
수준: 입문 단계
핵심 원칙: "코딩 전에 먼저 문서로 정의한다"
```

코드를 작성하기 전에 무엇을 만들 것인지를 먼저 문서로 정의합니다. 사양은 **참조 문서(Reference Document)** 의 역할입니다.

- 기능 개발 전에 spec.md를 먼저 작성
- 사양은 AI 대화에서 맥락 제공 용도로 활용
- 코드 완성 후 사양은 보관 또는 일부 업데이트

**적합한 상황**: 프로토타입, 단발성 기능, AI 코딩 어시스턴트 초기 도입 팀

> ⚠️ **한계**: 시간이 지나면서 코드와 명세의 드리프트(drift) 발생 가능

> **Level 1은 지금 당장 실천할 수 있는 가장 기본적인 SDD 형태입니다.**

---

#### Level 2: Spec-Anchored (명세 고정)

```
수준: 중급 단계
핵심 원칙: "사양이 개발의 기준점이다. 코드는 사양을 증명해야 한다"
```

사양이 개발의 기준점(Anchor)이 됩니다. 코드가 변경될 때마다 사양과의 일치 여부를 검증합니다.

```markdown
코드 리뷰 체크리스트:
  □ 모든 수용 기준(AC)에 대응하는 코드가 있는가?
  □ 엣지 케이스가 처리되었는가?
  □ 사양에 없는 기능이 무단으로 추가되지 않았는가?
```

**적합한 상황**: 대부분의 프로덕션 시스템, 장기 유지보수가 필요한 엔터프라이즈 프로젝트

**대표 도구**: Cucumber, Gherkin, GitHub Spec Kit

---

#### Level 3: Spec-as-Source (명세가 소스)

```
수준: 고급 단계
핵심 원칙: "사양이 코드의 소스다. 코드는 사양에서 생성된다"
```

사양 자체가 코드를 생성하는 소스입니다. AI는 사양을 입력받아 코드, 테스트, 문서를 모두 생성합니다.

```
사양 작성/수정 → AI가 해석 → 구현 코드, 단위 테스트, API 문서 자동 생성
                              → 개발자가 결과 검증 → 사양 정제
```

**적합한 상황**: 완전한 AI-Native 개발팀, 미션 크리티컬 시스템(항공, 의료, 금융), 멀티 에이전트 환경

**대표 도구**: AWS Kiro, GitHub Spec Kit, Google Antigravity

---

#### 3가지 레벨 비교

| 구분 | Level 1: Spec-First | Level 2: Spec-Anchored | Level 3: Spec-as-Source |
|---|---|---|---|
| **핵심 변화** | 코딩 전 문서화 | 코드가 사양을 증명 | 사양이 코드를 생성 |
| **사양의 역할** | 참조 문서 | 기준점 | 소스 |
| **도입 난이도** | 낮음 | 중간 | 높음 |
| **적용 대상** | 모든 팀 | 체계적인 팀 | 성숙한 팀 |

> **💡 실용적 조언**: Level 1부터 시작해서 팀의 역량과 함께 점진적으로 이동하는 것이 현실적입니다.

---

### 3.4 스펙과 Memory Bank: 두 가지 컨텍스트의 역할

AI와 효과적으로 협업하려면 두 가지 컨텍스트를 모두 제공해야 합니다.

```
AI가 필요로 하는 정보
  ├── [1] 스펙(Spec): "무엇을 만들어야 하는가?"
  │         → 기능별, 작업별로 달라지는 정보
  └── [2] Memory Bank: "어떤 맥락에서 만들어야 하는가?"
            → 프로젝트 전체에 걸쳐 일관되게 유지되는 정보
```

**스펙(Specification)**은 특정 기능에 대한 구체적인 요구사항으로, 새 기능 개발 시마다 작성됩니다.

**Memory Bank**는 프로젝트 전체에 적용되는 맥락 정보입니다. GitHub Spec Kit의 `constitution.md`, Cursor의 `.cursorrules`, Gemini의 `GEMINI.md`가 모두 Memory Bank의 구현 형태입니다.

```
[두 컨텍스트가 없을 때] → 매번 다른 스타일, 일관성 없는 코드
[스펙만 있을 때]        → 기능 구현되지만 기존 코드베이스와 스타일 불일치
[Memory Bank만 있을 때] → 프로젝트에 맞는 스타일이지만 엉뚱한 기능 구현
[스펙 + Memory Bank]    → 의도한 기능을 프로젝트 표준에 맞게 일관성 있게 구현 ✅
```

> **🔑 핵심**: 스펙은 기능을 정의하고, Memory Bank는 맥락을 제공합니다. 두 가지가 결합될 때 AI는 진정한 개발 파트너가 됩니다.

## 4. SDD의 4가지 핵심 원칙

### 원칙 1. 기본 아티팩트로서의 사양

사양은 진실의 중심 소스입니다. 코드는 사양을 표현한 결과물이며, 소프트웨어를 유지보수한다는 것은 **사양 자체를 발전시키는 것**을 의미합니다.

### 원칙 2. 실행 가능한 사양(Executable Specification)

사양은 작동하는 시스템을 생성할 수 있을 만큼 **정확하고, 완전하며, 명확**해야 합니다. "로그인 기능 만들어 줘"라는 지시는 SDD에서 사용자 스토리, 수용 기준, 요구 사항, 엣지 케이스로 구체화됩니다.

### 원칙 3. 살아있는 문서(Living Documentation)

- 버그를 수정한다는 것은 잘못된 코드를 만들어 낸 **사양을 수정**하는 것입니다.
- 리팩터링한다는 것은 명확성을 위해 **사양을 재구성**하는 것입니다.
- 사양은 구현과 항상 동기화된 상태를 유지합니다.

### 원칙 4. AI-인간 협업 구조

```
[기존 바이브 코딩]
  개발자 ──(모호한 지시)──▶ AI ──▶ 예측 불가능한 코드

[SDD + 에이전틱 엔지니어링]
  개발자 ──(명확한 사양)──▶ AI ──▶ 검증된 코드
     ▲          ▲                        │
  Memory    Spec 문서              피드백/검증
  Bank      (기능 명세)                  │
     └──────────┴────────────────────────┘
```

## 5. Less is More: 스펙 설계 전략

2절에서 "정보 과부하의 역설"을 배웠습니다. 그렇다면 **얼마나, 어떻게** 써야 할까요? 이 장에서는 효과적인 스펙을 설계하는 전략을 다룹니다.

---

### 5.1 Less is More 원칙이란?

> **"스펙의 가치는 길이가 아니라 밀도에 있다. 필요한 것만, 정확하게, 구조적으로."**

```
스펙 길이 vs AI 이해도:

짧고 핵심적인 스펙 → AI의 주의가 집중 → 높은 정확도
긴 비구조화 스펙   → AI의 주의가 분산 → 중요한 부분 누락

최적의 스펙: 충분히 구체적이되, 불필요한 내용 없이
```

Less is More 원칙의 실천 방향은 다음 세 가지입니다.

**첫째, JES(Just Enough Specification)**: 구현하기에 충분한 정보만. 구현 방법의 세부사항은 AI에게 위임합니다.

**둘째, 단일 책임 원칙**: 하나의 스펙 파일이 하나의 기능에만 집중합니다. 관련 없는 요구사항을 하나의 파일에 섞지 않습니다.

**셋째, 계층적 구조**: 범위에 따라 스펙을 계층으로 분리합니다. 프로젝트 전체 원칙과 개별 기능 명세를 같은 파일에 섞지 않습니다.

---

### 5.2 계층적 스펙 구조

효과적인 스펙 시스템은 범위(Scope)에 따라 **4개 레이어**로 구성됩니다.

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 0: 프로젝트 헌법 (constitution.md)                    │
│  "이 프로젝트는 이렇게 만든다"                               │
│  → 기술 스택, 보안 원칙, 코딩 컨벤션, 팀 규칙               │
│  → 변경 빈도: 매우 낮음 (분기 1회 이하)                     │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: 도메인/시스템 명세 (domain.md)                    │
│  "이 시스템은 어떤 도메인으로 구성되는가"                   │
│  → 주요 도메인, 서비스 간 관계, API 계약 개요               │
│  → 변경 빈도: 낮음 (새 도메인 추가 시)                      │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 기능 명세 (features/NNN-기능명/spec.md)           │
│  "이 기능은 이렇게 동작한다"                                │
│  → 사용자 스토리, 수용 기준, 엣지 케이스                   │
│  → 변경 빈도: 중간 (기능 변경 시)                           │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 구현 작업 명세 (features/NNN-기능명/tasks.md)     │
│  "이 기능을 어떤 순서로 구현한다"                           │
│  → 구체적 작업, 의존 관계, 완료 기준                        │
│  → 변경 빈도: 높음 (구현 진행에 따라)                       │
└─────────────────────────────────────────────────────────────┘
```

#### Layer 0: 프로젝트 헌법 (constitution.md)

프로젝트 전체에 적용되는 **변하지 않는 원칙들**입니다. 헌법의 규모는 **한 화면에 보일 정도**(약 50~80줄)가 적절합니다.

| 포함해야 할 것 (✅) | 포함하지 말아야 할 것 (❌) |
|---|---|
| 승인된 기술 스택과 버전 | 특정 기능의 요구사항 |
| 보안 원칙 (암호화 알고리즘 등) | 구현 세부사항 |
| 코딩 컨벤션 | 변경될 가능성이 있는 비즈니스 규칙 |
| 테스트 커버리지 기준 | 임시 결정사항 |

```yaml
# constitution.md 실제 예시
project_principles:
  architecture:
    - "모든 기능은 독립 라이브러리로 먼저 구현"
    - "마이크로서비스 간 직접 DB 접근 금지"

  quality:
    - "구현 코드 전 반드시 테스트 선작성 (TDD)"
    - "코드 커버리지 80% 미만 PR 자동 거부"

  ai_governance:
    - "AI 생성 코드는 반드시 결정적 엔진 검증 통과 필요"
    - "핵심 비즈니스 로직은 AI 직접 실행 금지"

  change_management:
    - "API 변경 시 반드시 Spec 먼저 업데이트"
    - "프로덕션 인시던트 → 명세 개선으로 반드시 반영"
```

#### Layer 2: 기능 명세 (spec.md) — 가장 자주 사용

특정 기능이 **무엇을, 왜** 해야 하는지를 정의합니다. AI가 코드를 생성할 때 가장 직접적으로 참조하는 레이어입니다.

```markdown
# 기능 명세: [기능 이름]
Feature-ID: NNN

## 사용자 스토리
[역할]로서, [목표]를 위해 [기능]을 원한다.

## 수용 기준 (Acceptance Criteria)
- Given: [전제 조건]
  시나리오가 시작되기 전 상태
  . 사용자 상태
  . 시스템 상태
  . 데이터 상태
  “이 상황이 성립하려면 무엇이 이미 준비되어 있어야 하지?”

  When:  [사용자 행동]
  사용자가 하는 행동 또는 시스템 이벤트
  . 클릭
  . 입력
  . 요청 전송
  “사용자가 무엇을 했을 때?”

  Then:  [예상 결과]
  행동 이후 발생해야 하는 결과
  . 화면 변화
  . 메시지
  . 데이터 저장
  “그래서 무엇이 보여야 / 일어나야 하지?”

  “이 상태에서(Given) → 이 행동을 하면(When) → 이런 결과가 나와야 한다(Then)”

## 기능 요구사항
- MUST:   [반드시 구현해야 함]
  (필수 요구사항)
  . 사용자는 PDF 파일을 업로드할 수 있어야 한다
  . 업로드 성공 시 완료 메시지를 표시해야 한다
  . 제출 시각을 기록하고 사용자에게 표시해야 한다

- SHOULD: [가능하면 구현]
  (권장 요구사항)
  . 업로드 진행 상태(progress)를 표시한다
  . 동일 파일 재업로드 시 경고 메시지를 제공한다

- MAY: [선택적 구현]
  . 드래그 앤 드롭 업로드 지원
  . 업로드 히스토리 조회 기능

## 비기능 요구사항
- 성능: [측정 가능한 기준]
  . 파일 업로드 응답 시간은 3초 이내 (10MB 기준)
  . 동시 사용자 100명 이상 처리 가능해야 함

- 보안: [보안 제약사항]
  . 인증된 사용자만 업로드 가능해야 함
  . 파일 타입 검증 (PDF만 허용)
  . 업로드 파일에 대한 악성 코드 검사 수행

- 가용성
  . 업로드 실패 시 재시도 가능해야 함

## 엣지 케이스
정상적인 흐름이 아닌 “경계 상황 / 예외 상황”에서의 동작을 정의한 것
엣지 케이스는 단순 나열이 아니라 **이 상황에서 어떻게 처리할지까지 정의**해야 함

- [예외 상황 1]: [처리 방법]
- [예외 상황 2]: [처리 방법]
  . 파일이 없는 경우 : 에러 메시지 표시
  . 10MB 초과 파일 : 업로드 제한 안내
  . 네트워크 끊김 : 업로드 실패 및 재시도 안내
  . 동일 파일 중복 제출 : 정책에 따라 허용/차단

## ❌ How는 여기서 작성하지 않는다 (구현 방법은 plan.md에)
```

### 5.3 스펙 작성 실전 가이드: 좋은 스펙과 나쁜 스펙

---

#### 나쁜 패턴 1: 모호한 형용사 사용

```markdown
❌ 나쁜 스펙:
"검색 결과는 빠르게 반환되어야 한다."
"사용자 인터페이스가 직관적이어야 한다."

→ 문제: "빠르게", "직관적으로"는 AI가 수치화할 수 없다.
```

```markdown
✅ 좋은 스펙:
"검색 결과는 500ms 이내에 반환되어야 한다.
"모바일 화면(375px 이상)에서 버튼의 터치 영역이 44px 이상이어야 한다."
```

---

#### 나쁜 패턴 2: 구현 방법을 지시하는 스펙

```markdown
❌ 나쁜 스펙:
"React의 useState 훅을 사용하여 로그인 상태를 관리한다."
"MySQL의 InnoDB 엔진을 사용하고 인덱스를 username 컬럼에 추가한다."

→ 문제: 이것은 "어떻게(How)"에 해당한다. 스펙은 "무엇을(What)"에 집중해야 한다.
```

```markdown
✅ 좋은 스펙:
"사용자의 로그인 상태는 페이지 이동 시에도 유지되어야 한다."
"사용자 검색 성능: 100만 건 데이터에서 1초 이내 결과 반환"
```

---

#### 나쁜 패턴 3: 검증 불가능한 요구사항

```markdown
❌ 나쁜 스펙:
```

## 6. SDD 4단계 워크플로

```
[1단계: 지정] ──▶ [2단계: 계획] ──▶ [3단계: 작업] ──▶ [4단계: 구현]
  Specify              Plan              Task             Implement
  "무엇을?"           "어떻게?"        "어떤 순서로?"      "코드 생성"
```

각 단계는 다음 단계의 입력물이 되는 아티팩트를 생성하며, 요구 사항에서 최종 코드까지의 완전한 추적성(Traceability)이 보장됩니다.

> 이 4단계가 GitHub Spec Kit의 `/speckit.specify` → `/speckit.plan` → `/speckit.tasks` → `/speckit.implement` 명령어와 1:1 대응됩니다.

---

### 1단계: 지정(Specify) — What & Why

**"소프트웨어가 무엇을 해야 하는지, 그리고 왜 해야 하는지를 정의합니다."**

#### 사용자 스토리(User Story)

```
형식: "나는 [사용자 유형]으로서, [목표]를 달성하기 위해 [기능]을 원한다."
예시: "나는 쇼핑몰 회원으로서, 주문 내역을 한눈에 확인하기 위해
       마이페이지에서 최근 주문 목록을 조회하고 싶다."
```

#### 수용 기준(Acceptance Criteria)

```
Given: 로그인한 사용자가 마이페이지에 접속했을 때
When:  주문 내역 탭을 클릭하면
Then:  최근 3개월 이내 주문 목록이 날짜 내림차순으로 표시된다.
```

#### 요구 사항(Requirements)

**기능적 요구 사항(Functional Requirements)**

- `MUST`: 회원가입 시 이메일 중복 여부를 확인해야 한다.
- `MUST`: 비밀번호는 최소 8자 이상, 영문·숫자·특수문자를 각각 1개 이상 포함해야 한다.
- `SHOULD`: 주문 완료 후 사용자에게 확인 이메일을 발송해야 한다.

**비기능적 요구 사항(Non-Functional Requirements)**

- 성능: 주문 목록 조회 응답 시간은 2초 이내여야 한다.
- 보안: 개인정보는 AES-256 암호화로 저장해야 한다.
- 가용성: 시스템 가동률은 99.9% 이상을 유지해야 한다.

#### 엣지 케이스(Edge Cases)

- **경계값**: 주문 수량이 0개일 때, 재고가 정확히 1개 남았을 때
- **빈 값 및 null**: 검색어 없이 검색 버튼을 눌렀을 때
- **권한 예외**: 로그인하지 않은 사용자가 마이페이지에 접근할 때
- **동시성**: 두 사용자가 동시에 마지막 재고 1개를 구매하려 할 때

### 2단계: 계획(Plan) — How

**"어떻게 구축할 것인지를 결정합니다."**

```markdown
# Implementation Plan: 003-realtime-chat

## 기술 선택 및 근거
| 기술        | 용도                     | 근거                                     |
|-------------|--------------------------|------------------------------------------|
| WebSocket   | 실시간 양방향 통신       | HTTP Long-polling 대비 70% 오버헤드 감소 |
| PostgreSQL  | 메시지 이력 영구 저장    | ACID 보장, 기존 인프라와 통합            |
| Redis       | 사용자 Presence 상태 관리 | TTL 지원으로 자동 오프라인 처리         |

## API 계약
WebSocket Events:
  · message:send    → message:receive
  · presence:update → presence:broadcast

REST:
  · GET  /api/rooms/{roomId}/messages?cursor=...&limit=50
  · POST /api/rooms/{roomId}/messages
```

### 3단계: 작업(Task) — When & Who

**"계획을 구체적이고 실행 가능한 개발 작업 단위로 분해합니다."**

좋은 작업의 크기는 **하루 이내에 완료할 수 있는 수준**이 이상적입니다.

```
[계획] 회원가입 기능 구현
  ├── [P] 작업 1: User 데이터 모델 및 DB 스키마 생성  (병렬 가능)
  ├── [P] 작업 2: 이메일 중복 확인 API 구현           (병렬 가능)
  ├── [P] 작업 3: 비밀번호 유효성 검증 로직 구현      (병렬 가능)
  ├── [ ] 작업 4: 회원가입 API 엔드포인트 구현        (1,2,3 완료 후)
  ├── [ ] 작업 5: 가입 완료 확인 이메일 발송 구현     (4 완료 후)
  └── [ ] 작업 6: 단위 테스트 작성                   (5 완료 후)

[P] = 병렬 실행 가능   [ ] = 선행 작업 완료 후 실행
```

### 4단계: 구현(Implement) — Code

**"사양, 계획, 작업 목록을 기반으로 코드를 작성합니다."**

```
[작업 선택] ──▶ [사양 확인] ──▶ [코드 생성] ──▶ [수용 기준 검증] ──▶ [다음 작업]
                                                         │
                                        (실패 시 사양 재검토 또는 코드 수정)
```

추적성(Traceability) 확보 예시입니다.

```python

# Acceptance Criteria: AC-12-1, AC-12-2
# Task: TASK-045
def get_order_list(user_id: str) -> list:
    ...
```

### Context Engineering과 SDD

> **프롬프트 엔지니어링**: "어떻게 말하느냐" (How to say)
> **Context Engineering**: "무엇을 알려주느냐" (What to tell)

```
낮은 맥락: "적 AI 만들어줘"
  → Update()에서 기본 추적/이동 코드 생성 (품질 낮음)

높은 맥락: "우리 프로젝트는 NavMesh 기반 길찾기를 씀.
            적 상태 관리는 FSM으로 하고 있음."
  → 프로젝트 아키텍처와 완전히 일치하는 정교한 코드 생성
```

**SDD = 구조화된 Context Engineering:**

```
Constitution (프로젝트 불변 원칙)      ← 가장 상위 맥락
      +
Spec (기능 요구사항 + 비즈니스 목표)   ← 기능 수준 맥락
      +
Plan (기술 설계 + API 설계)            ← 구현 수준 맥락
      +
Tasks (실행 단위 + 병렬화 정보)        ← 실행 수준 맥락
      ↓
Context Rot 원천 방지  ✅
```

### 7.3 Böckeler가 던진 핵심 질문들

#### 질문 1: "하나의 워크플로우로 모든 크기의 문제를 커버할 수 있는가?"

```
소규모(< 30분): 자유 형식 지시 → 코드 수정 → 완료
중규모(30분~1일): 간략한 spec → 구현
대규모(1일 이상): 전체 SDD 워크플로 적용
```

#### 질문 2: "스펙 리뷰가 코드 리뷰보다 나은가?"

```
스펙 리뷰가 잡아내는 것:   요구사항 누락, 설계 방향 오류, 논리 오류
코드 리뷰가 잡아내는 것:   구현 수준의 버그, 성능 문제, 컨벤션 위반
```

→ 두 리뷰는 대체 관계가 아니라 **보완** 관계입니다.

#### 질문 3: "통제의 착각은 아닌가?"

스펙은 **필요 조건**이지 **충분 조건**이 아닙니다.

```
착각: 스펙 작성 → AI가 스펙대로 코드 생성 → 완성
현실: 스펙 작성 → AI가 스펙을 나름대로 해석 → 반드시 검증 필요
```

#### 질문 4: "MDD의 실패를 반복하는 것은 아닌가?"

| 비교 항목 | MDD | SDD |
|---|---|---|
| 스펙 언어 | 형식적 모델링 언어(UML, DSL) | 자연어 + 구조화된 마크다운 |
| 코드 생성 | 규칙 기반 컴파일러 | 자연어 이해 기반 LLM |
| 사람의 참여 | 최소화 지향 | 검증 단계에 필수 참여 |

**SDD가 MDD처럼 실패하는 패턴 방지 원칙**: "코드를 수정하기 전에 반드시 스펙을 먼저 수정한다."

## 8. 엔터프라이즈 레거시 전환에서의 SDD 적용

### 8.1 레거시 시스템의 현실

```
레거시 시스템의 전형적인 상태:
  ├── 문서 없음 또는 현행화 되지 않은 문서
  ├── 원래 개발자가 이직하여 맥락 상실
  ├── 코드만이 "진실"이나 코드도 이해하기 어려움
  └── 새 기능 추가가 두려운 상태 ("무너질까봐")
```

#### SDD가 레거시 전환의 리스크를 줄이는 방법

```
[기존 접근법 (코드 → 코드)]
레거시 코드 → AI가 분석 → 새 코드 생성
                ↑ 비즈니스 규칙 누락 위험 매우 높음

[SDD 접근법 (코드 → 스펙 → 코드)]
레거시 코드 → 역분석하여 스펙 도출 → 스펙 검증 → 새 코드 생성
```

### 8.2 기존 산출물에서 스펙 도출하는 5단계 프로세스

**1단계**: 기존 문서 분류 및 신뢰도 평가 

**2단계**: AI를 활용하여 기존 코드에서 비즈니스 규칙 추출

```
AI 프롬프트 예시:
"아래 코드에서 비즈니스 규칙만 추출하여 Given-When-Then 형식으로
 정리해줘. 구현 방법은 제외하고, '사용자 관점에서 어떤 일이 발생하는가'에만 집중해줘."
```

**3단계**: 중복 및 모순 제거

**4단계**: 도메인 전문가(현업 담당자) 검증

**5단계**: SDD 구조(Layer 0~3)로 정리 및 테스트 작성

```
테스트 원칙:
  "레거시 시스템에서 통과하는 테스트가 새 시스템에서도 반드시 통과해야 한다."
```

### 8.3 점진적 전환 전략: Big Bang은 실패한다

```
성공하는 전환 전략 — 4단계:

① 20/80 규칙: 상위 20%의 고위험 기능에 먼저 SDD 적용
② 파일럿: 3~6명 소규모 팀, 8~12주 파일럿 진행
③ 병행운영:
   Phase 1 (1~3개월): 파일럿 팀만 SDD
   Phase 2 (3~6개월): 자원하는 팀에게 확대
   Phase 3 (6~12개월): 조직 전체 적용
④ 측정: 버그 발생률, Lead Time, 팀 만족도 추적
```

## 9. SDD와 AI-Native 설계 패러다임의 연결

### 9.1 규칙기반 → 의도기반으로의 전환

```python
# ❌ 기존 규칙기반 설계
if language == "ko":
    route_to(korean_handler)
elif language == "en":
    route_to(english_handler)
# 새 언어 추가 시마다 코드 수정 필요

# ✅ AI-Native 의도기반 설계
spec = """
입력의 언어, 맥락, 의도를 해석하여 적절한 처리 경로를 반환하라.
출력 형식: {"language": "...", "intent": "...", "confidence": 0.0~1.0}
"""
result = llm.interpret(user_input, spec)
route_by_intent(result)
# 새 언어/의도 추가 시 명세(Spec)만 업데이트
```

핵심 원칙: **"AI는 방향을 제시하고, 시스템은 정확하게 수행한다."**

잘못된 구조 : LLM → "결제 진행" → 실제 결제 실행

올바른 구조 : LLM → "이건 일반 주문이다" (판단) → Backend 시스템 → 결제 실행

### 9.2 AI 능력과 스펙의 가치: 비례 관계

흥미로운 역설이 있습니다. AI가 더 강력해질수록 스펙의 중요성은 줄어드는 것이 아니라 **더 커집니다.**

> **비유**
> 자전거는 느리지만 방향이 틀려도 쉽게 돌아올 수 있습니다.
> 고속열차는 빠르지만, 방향이 틀리면 돌이키기 어렵습니다.
> AI가 강력해질수록 고속열차와 같아집니다. **스펙은 열차의 레일(Rail)입니다.**

### 9.3 SDD의 위치: 현재와 미래 방향

```
현재 (2026):   Human → Spec → AI → Code → Human Verification
가까운 미래:   Human → Spec → AI → Code → AI Verification → Human Approval
먼 미래:       Human → Intent → AI → Spec → AI → Code → AI Verification → Human Approval
```

변하지 않는 것:
- 스펙이 AI 협업의 핵심 인터페이스라는 사실
- 인간이 의도(Intent)와 가치(Value)를 정의해야 한다는 사실
- 검증과 최종 승인에 인간의 판단이 필요하다는 사실

## 10. Anti-Pattern: 피해야 할 것들

### ❌ Anti-Pattern 1. AI를 핵심 비즈니스 로직에 직접 연결

```
❌ 사용자 주문 → LLM이 직접 결제 처리
   이유: LLM은 확률적. 1%의 오류 = 하루 100만 건 중 1,000건 실패

✅ 사용자 주문 → LLM이 "이건 일반 주문" 판단 (확률적)
               → 결정적 엔진이 결제 처리 (결정론적)
```

### ❌ Anti-Pattern 2. 구조 없는 원시 AI 생성 (Vibe Coding)

```
❌ 구조없는 단순한 요구 → "“로그인 기능 만들어줘”
   결과 : 매번 결과가 다름, 코드 스타일 AI마다 제각각, 재현 불가능, 유지보수 지옥(AI가 감으로 코딩하는 것)
   영향도 : 팀 개발 불가능, 코드 일관성 붕괴, 품질 통제 불가

✅ Constitution + Spec + Plan → AI 코드 생성
   = 항상 동일한 맥락에서 일관된 코드 생성
   . Constitution → 팀 규칙 (코딩 스타일, 원칙)
   . Spec → 무엇을 만들 것인가
   . Plan → 어떻게 만들 것인가

   **AI는 맥락이 있어야 제대로 동작한다**
```

### ❌ Anti-Pattern 3. 명세 없는 AI 코드 병합

```
❌ AI가 만든 코드 → 바로 Merge
  . 검증 없음
  . 테스트 없음
  . 기준 없음
  결과 : 버그 유입 가능성, 품질 붕괴 가능성 높음

✅ AI 코드 생성 → 검증 → 수용 기준 테스트 통과 → 승인 후 Merge

  **Spec = 코드의 기준**
```

### ❌ Anti-Pattern 4. 명세 유지 없이 코드만 수정 (Spec Drift)

```
❌ 코드만 수정, Spec은 그대로
  결과 : AI가 잘못된 Spec 기반으로 코드 생성
        팀 커뮤니케이션 붕괴

✅ 기능 변경 시 → 명세 먼저 수정 → AI가 코드 재생성
```

### ❌ Anti-Pattern 5. 너무 상세한 스펙으로 AI를 과도하게 제약

```
❌ 너무 상세한 스펙 : 모든 구현까지 다 적음, 예: 변수명, 로직, 구조까지 지정
   결과 : AI가 창의적으로 최적화 못함, Spec 유지보수 어려움, 변화 대응 느림

✅ 스펙(spec.md): What + Why만
   계획(plan.md): How (기술 선택, API 설계)
   작업(tasks.md): When + Who (순서, 병렬화)

   **Spec은 방향만 제시하고, 구현은 맡겨라.**
```

## 11. 미래 전망: AI-Native SDLC(Software Development Life Cycle)

### SDD와 TDD의 역사적 유사성

```
TDD의 교훈:
  → 모든 팀이 TDD를 하는가?  아니다.
  → 테스트를 안 짜는 팀이 있는가?  거의 없다.
  → TDD는 방법론이지만, "테스트"는 업계 표준이 됨.

SDD의 미래 예측:
  → SDD를 안 해도 "Spec"은 쓰게 될 것이다
  → AI가 아무리 발전해도, 맥락(Context) 없이는 좋은 코드를 짤 수 없다.
```

### 개발자 역할의 변화

```
Before AI-Native:
  개발자 = 코드 작성자
  주요 스킬: 프로그래밍 언어, 알고리즘, 디버깅

After AI-Native (SDD 시대):
  개발자 = 명세 설계자 + 맥락 엔지니어 + AI 검증자
  주요 스킬:
    · 비즈니스 요구사항을 명확한 명세로 변환하는 능력
    · AI 출력을 검증하고 품질을 보장하는 능력
    · Context Engineering — 구조화된 맥락 설계 능력
      맥락없는 스펙 : "로그인 만들어줘" 
      구조화된 맥락 
        목표: 학생 과제 제출 시스템 로그인 기능

        환경:
          - React + Node.js
          - JWT 인증

        요구사항:
          - 이메일/비밀번호 로그인
          - 로그인 실패 시 에러 메시지
          - 로그인 성공 시 토큰 발급

        제약:
          - 보안 필수
          - 세션 사용 금지
      Prompt는 “질문”, Context Engineering은 “환경/구조 설계”
```

> **🔑 최종 메시지**
>
> AI가 코드를 생성하는 시대에서, 개발자의 가장 중요한 역량은 **"AI가 이해할 수 있는 언어로 의도를 정확하게 표현하는 능력"** 입니다.
>
> SDD는 그 언어를 배우고 실천하는 방법론입니다.
>
> AI 시대에도 개발자는 사라지지 않는다. 역할이 바뀔 뿐이다.

## 12. 정리 및 핵심 교훈

```
바이브 코딩  →  AI 코딩의 현실  →  SDD  →  에이전틱 엔지니어링  →  AI-Native

```

**핵심 공식:**

```
전통 개발:        Code → Docs
TDD:              Test → Code
SDD + AI-Native:  Spec → Code (AI 생성)
```

## 13. 빠른 참조 카드

### ⚡ SDD 단계별 체크리스트

**명세 작성 전:**
- [ ] Constitution(헌법)이 정의되어 있는가?
- [ ] Feature-ID가 부여되었는가?

**명세 작성 중 (Specify):**
- [ ] `[NEEDS CLARIFICATION]` 태그가 모두 해소되었는가?
- [ ] Acceptance Criteria가 측정 가능한 형태인가?
- [ ] How(어떻게)가 Spec 단계에 섞여 있지 않은가?

**구현 후:**
- [ ] 수용 기준 테스트 모두 통과?
- [ ] 헌법 원칙 준수 확인?
- [ ] 스펙과 코드가 동기화되어 있는가?

---

### 📚 핵심 용어 사전

| 용어 | 설명 |
|---|---|
| **SDD** | Specification-Driven Development. 명세를 1차 산출물로 삼는 개발 방법론 |
| **SSoT** | Single Source of Truth. 명세가 유일한 진실의 원천 |
| **Vibe Coding** | 명세 없이 즉흥적 프롬프트로 코드를 생성하는 방식 |
| **Context Rot** | 작업 전환 시 AI의 맥락이 증발하여 코드 일관성이 깨지는 현상 |
| **Memory Bank** | 프로젝트 전체에 걸쳐 일관되게 유지되는 맥락 정보 (constitution.md 등) |
| **Spec Drift** | 코드가 변경되었지만 명세가 업데이트되지 않아 발생하는 불일치 |
| **Context Engineering** | AI에게 구조화된 맥락을 제공하여 출력 품질을 높이는 기법 |
| **Agentic Engineering** | AI 에이전트를 조율하고 감독하는 전문 개발 방식 |
